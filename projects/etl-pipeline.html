Building an Omnichannel Retail Data Pipeline

End-to-End ETL · Star Schema · RFM Segmentation

I built an ETL pipeline to process millions of sales records for a fast-growing technology retail chain. Driven by a custom Python orchestrator, the system cleans raw data and calculates customer segments — delivering an analysis-ready Data Warehouse for the analytics team.

📅 2024–2025 🛠️ Python, BigQuery, GCS, Power BI 📊 6M+ Records

01. Business Challenges

A fast-growing technology retail chain — selling through both Shopify storefronts and direct online order systems — had no unified view of its own business.

Every month, analysts had to manually pull data from multiple sources, reconcile mismatched IDs, and stitch together reports in spreadsheets.

🗄️ Data Silos

Shopify, Online Orders, ZaloPay, MoMo, PayPal, and Mercury Bank — all storing data independently.

👤 No Customer View

Tracking a single customer's journey across channels was practically impossible.

📋 Manual Reporting

Cashflow and sales reports were compiled by hand each month — slow, error-prone, and inconsistent.

02. My Approach

I prioritised getting the data architecture right first — so any downstream tool could plug in reliably.

1

Ingest

Raw data dumps ingested into Google Cloud Storage buckets as the single source of truth.

2

Transform

Python scripts decompress, parse, clean, and standardise into a unified schema with data quality checks.

3

Load & Segment

Transformed data loaded into BigQuery. SQL MERGE calculates and updates each customer's RFM segment.

03. System Architecture

The pipeline follows a cloud-native approach: raw files land in GCS, get processed by a custom Python orchestrator, and are loaded into BigQuery.

📐 Architecture Diagram

Figure 1 — High-level pipeline architecture: from raw sources to analysis-ready Data Warehouse.

How the Pipeline Works

Data flows through three layers — the Medallion Architecture:

🥉 Bronze — Raw

Compressed files ingested and stored as-is in GCS.

🥈 Silver — Cleaned

Python standardises schemas, fixes data types, generates surrogate keys.

🥇 Gold — Serving

Dimension & Fact tables loaded into BigQuery with SQL views for reporting.

04. Results

The team gains a centralised, analysis-ready Data Warehouse in BigQuery:

4.1 — Final Data Structure

A Star Schema data warehouse where all events connect back to Customers and Products.

Dimension Tables

  • dim_customer — profile + RFM segment
  • dim_product — SKU, category, brand

Fact Tables

  • fact_orders — all orders
  • fact_order_items — line detail
  • fact_payment — e-wallet
  • fact_bank_transactions — bank
  • fact_cart_events — behaviour

4.2 — Auto-Updated Customer Metrics

The pipeline recalculates LTV, last order date, order count, and average spend — re-classifying each customer into an RFM-based segment (11 segments).

4.3 — Reporting Views

📊 Daily Cashflow View

Reconciles orders, payments, and bank movements into one daily summary.

🛒 Customer Journey View

Stitches cart events and purchases into a chronological sequence per customer.

05. Business Impact

With the pipeline in place, here is what changes for the business:

🎯 One Source of Truth

All data now lives in one place — clean, consistent, and version-controlled.

🔌 Plug-and-Play Analytics

The team connects directly to BigQuery — no manual data prep needed.

💰 Daily Financial Visibility

Finance reconciles all transactions into one daily cashflow summary.

👥 Always-Current Customer View

Marketing sees Champions, Loyals, and At-Risk customers — updated automatically.
